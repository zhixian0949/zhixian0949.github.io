深入理解Spring MVC思想

2021年03月03日 Guestbook

工作流程

  1. 用户发送请求 http://localhost:9080/springmvc-chapter2/hello，web容器接收到请求，根据在web.xml中配置的拦截规则/，/hello会映射到DispatchServlet处理。
  2. DispatchServlet会将请求匹配到对应的Bean去处理。默认由BeanNameUrlHandlerMapping来作此映射，对应的Bean就是注解@Controller所标记的类里的方法。BeanNameUrlHandlerMapping会将这个Bean包装成HandlerExecutionChain，要是有拦截器，拦截器也在此Chain中，没有则只包括Handler。
  3. DispatchServlet会将Chain中的处理器适配为SimpleControllerHandlerAdapter。
  4. SimpleControllerAdapter会调用Handler中对应的方法进行处理，最终返回ModelAndView给DispatchServlet。
  5. 视图解析器会根据ModelAndView里的逻辑视图去找物理视图，/WEB-INF/jsp/hello.jsp。
  6. 视图渲染会将ModelAndView中的数据在渲染到物理视图中
  7. 最终返回给DispatchServlet，DS再响应给浏览器。

此处涉及到：

  • DispatchServlet
  • HandlerMapping
  • HandlerAdapter
  • ViewResolver
  • 视图解析器
  • 视图

DispatchServlet

  • 功能：

    1. 文件上传解析
    2. 通过HandlerMapping将请求映射到具体的处理器，返回一个HandlerExecutionChain，包括处理器和拦截器
    3. 通过HandlerAdapter支持多类型的处理器
    4. 通过ViewResolver将逻辑视图解析为物理视图
    5. 渲染视图
    6. 遇到异常交给HandlerExceptionResolver解析

  • 在web.xml中的配置

    • 第一个问题是springmvc配置文件的问题

      1、不写,使用默认值:/WEB-INF/-servlet.xml
      2、/WEB-INF/classes/springMVC.xml
      3、classpath*:springMVC-mvc.xml
      4、多个值用逗号分隔

    • 第二个问题便是拦截规则问题

      1. *.do,*.html:不会导致静态文件（jsp，css）被拦截
      2. /REST风格，比较流行，会导致静态文件被拦截，但是有解决方法
      3. /*会执行Handler，但是到jsp时会再次被拦截，不能访问jsp

    • 第三个问题是父子上下文的关系

      1. 传统型：父上下文保存服务层、DAO层、事务的Bean，子上下文保存处理器的Bean，子上下文可以访问父上下文，反之，父不能访问子
      2. 激进型：想要快速开发，没有乱七八糟的分层，服务层移到处理器层，不需要父上下文

  • 如何访问到静态文件

    1. 激活Tomcat的defaultservlet来处理静态文件

    2. ```xml

      <mvc:resources mapping=”/images/**” location=”/images/“ />

《Java编程思想内部类》

2021年01月22日 Guestbook

创建内部类

  • 将类定义在外部类内部即可

链接到外部类

  • 构建内部类对象时，需要一个指向其外部类对象的引用，如果编译器访问不到这个引用就会报错

使用.this和.new

  • 在内部类中可以通过Outer.this返回外部类对象的引用，在其他类中需要创建内部类对象时，可以通过外部类对象.new 内部类来创建

内部类向上转型

  • 当内部类私有时，不能访问其名字，只能转型为其基类

局部内部类

  • 内部类可以放入方法中，这样这个内部类属于方法，不属于外部类。在此涉及到一个问题：方法的生命周期和局部内部类的声明周期不一样。方法结束后，参数就消失了，此时局部内部类对象依旧可以生存，所以在局部内部类中使用局部变量实际上是用的局部变量的一个拷贝。这样又衍生出一个问题，局部变量修改了以后，他的拷贝不能同时修改，会造成数据不一致的问题。为了解决这个问题，jdk规定，在局部内部类中使用的局部变量必须是final的，虽然jdk8没此规定，但是在编译时是悄咪咪的加上去了的。

匿名内部类

  • 匿名内部类不可能有构造器，可以通过代码块赋值

继承内部类

  • 内部类对象会持有外部类对象的引用，但是内部类的子类并不会持有这个引用，所以构造函数需要特殊处理

深入理解Java虚拟机脉络

2021年01月10日 Guestbook

深入理解Java虚拟机脉络

一、Java发展史

  • 96年1月，发布JDK1.0 。特性：Java虚拟机、Applet、AWT
  • 97年2月，发布JDK1.1 。特性：JAR文件格式、JDBC、JavaBeans、RMI等，内部类、反射
  • 98年12月，发布JDK1.2 。将Java拆分为3个体系，J2SE、J2EE、J2ME 。第一次内置了JIT即时编译器。同时新增了strictfp关键字和Collection的API
  • 99年4月，HotSpot虚拟机诞生，JDK1.3以后，他成为了默认虚拟机
  • 00年5月，JDK1.3发布。新增数学运算的类库，JNDI
  • 02年发布JDK1.4 。特性：正则表达式、NIO、日志类、XML解析器
  • 04年，JDK1.5发布。自动装箱、泛型、动态注解、枚举、可变长参数、循环遍历。同时改进了Java内存模型，提供了concurrent并发包
  • 06年JDK1.6发布，对虚拟机内部进行了大量的改进：包括锁和同步、垃圾回收、类加载
  • 09年发布JDK7 。

二、内存区域划分

  1. 程序计数器：
    • 存放的是将要执行的指令的行号
    • 线程私有，每个线程都有一个程序计数器
    • 如果执行的是本地方法，则内容为空
    • 唯一一个不会发生OOM的区域
  2. Java虚拟机栈
    • 线程私有
    • 数据结构：栈帧
    • 栈帧：局部变量表、操作数栈、动态连接、方法出口
    • 局部变量表：8种基本数据类型、reference对象引用、returnAddress。基本单位为槽，一个槽32位，double和long类型数据占用2个槽。一个局部变量表的大小在编译器便确定了。
    • 操作数栈：可以存放各种数据类型，一个栈深32位，double和long占2个栈深；编译时栈的最大深度就可以确定了；运行过程中各个指令会做进栈出栈的操作；该方法调用其他有返回结果的方法时，其他方法结束时会将结果push到该操作数栈上
    • 动态连接：指向运行时常量池中的方法引用
    • 方法出口：返回地址
    • 若线程需要的虚拟机栈深度大于虚拟机所允许的最大深度，则抛出StackOverflowError。
    • 若允许虚拟机栈动态扩展，当无法申请到足够的空间时抛出OOM
  3. 本地方法栈
    • 与虚拟机栈功能一样。
  4. Java堆
    • 线程共享
    • 分为Eden区，From Survivor区、To Survivor区、老年代
    • 目前为止，对象均在堆上分配。随着逃逸分析越来越强大，可以在栈上分配
    • 并发分配时：CAS和TABL保证线程安全
  5. 方法区：
    • 在JDK1.6及更早，实现为永久代。
    • JDK1.7，将字符串常量池和静态变量保存到堆上
    • JDK1.8取消永久代，在本地内存实现元空间，将永久代中其他的内容移到元空间
    • 运行时常量池：类文件常量池在加载后会移到运行时常量池，其还有动态性，运行时可添加，如string.intern
  6. 直接内存
    • NIO缓存会用到此内存区域，避免本地内存和堆内存的频繁交换数据
    • XX:MaxDirectMemorySize指定，不指定默认与堆最大值一样（-Xmx）
    • 分配内存时要给此区域预留内存

三、Hotspot对象探究

  1. 对象的创建

    • 遇到new指令，先检查该指令参数在常量池中找到一个类的符号引用
    • 检查该类是否加载、解析、初始化
    • 如果没有，需要加载
    • 分配内存
      • 指针碰撞
      • 空闲列表
    • 并发问题：对象A正在分配内存，还没来得及修改指针，对象B就开始创建。两个对象的内存区域存在冲突
      • CAS
      • TABL：-XX:+/-UseTLAB
    • 初始化为初始变量值0，0.0，false等等
    • 设置对象头
    • 执行构造方法

  2. 对象的内存布局

    • 对象头

      • 类本身信息：一个动态数据结构，根据标志位来确定选用哪一种数据结构

      • 类型指针，表明对象属于那个类

      • 如果是数组，还有数组长度信息

    • 实例数据

    • 对其填充：8字节对其

  3. 对象访问定位

    • 句柄
    • 直接指针：HotSpot使用

四、垃圾回收

  1. 垃圾回收涉及的区域
    • 堆
    • 方法区
  2. 判断对象存活否
    • 引用计数法：循环引用时无法回收
    • 可达性分析：GCRoot
      • 虚拟机栈中引用的对象
      • 方法区静态属性引用的对象
      • 方法区常量引用的对象
      • 本地方法JNI引用的对象
  3. 引用分类
    • 强引用：永远不会回收
    • 软引用：内存不够时回收
    • 弱引用：存活到下一次回收
    • 虚引用：没有丝毫影响，只为了在对象被回收时可以收到一个系统通知
  4. 方法区的回收
    • 意义不大
    • 回收两类内容：废弃常量、无用的类
    • 判定无用类的条件：
      • 该类的所有实例被回收
      • 加载器被回收
      • 该类的java.lang.Class对象不被引用
  5. 垃圾回收算法
    • 标记-清除
      • 标记出需要清理的算法，然后清除。或者相反
      • 效率不稳定：需要大量标记和清除过程
      • 空间碎片化
    • 标记-复制
      • 将区域分成两块
      • 将存活的对象复制到另一边，然后清空
      • 缺点：浪费空间
    • 标记-整理
      • 将存活对象移至一端，清除另一端
      • 过程中需要需要暂停程序
      • 可以采取“和稀泥”的方式，暂时容忍碎片的存在，采取标记-清理。实在不行采取标记-清理。
    • 分代收集
      • 前提假设：生存周期越长，越不容易被回收
      • 所以按年龄分成不同的区域，采取不同的回收策略
  6. 内存分配和回收策略
    • GC
      • Minor GC：负责清理Eden和Surviver。采取标记-复制的算法。会触发stop-the-world。
      • Major GC：清理老年代
      • Full GC：清理年轻代和老年代
    • 空间分配担保
      • 对象优先在Eden分配
      • 大对象直接放入老年代，大对象的标准可通过参数设置
      • 长期存活的对象进入老年代，默认为年龄15
      • 动态年龄判断，如果Survivor区提前达到占用一半的要求，则不必等到15
      • Minor GC之前，虚拟机必须检查老年代最大可用连续空间必须大于新生代所有对象的总和。成立则安全
      • 若不成立，检查是否允许担保失败，若允许，则检查老年代最大连续可用空间是否大于历次晋升老年代对象的平均大小，大于则Minor GC
      • 小于或不允许失败，则Full GC

五、类文件结构

  1. Class文件结构

    • 概述

      • Class文件是一字节为基础的二进制流

      • 各数据之间没有分隔符

      • 大于1字节的数据按大端法存储

      • 整体文件的数据结构类似C中的结构体

      • 结构体中只存在两种数据结构：无符号数和表

    • 魔数：确定是class文件

    • 版本号：JDK1对应45，高版本虚拟机可以执行低版本文件，拒绝执行更高版本文件

    • 常量池：

      • 大小不固定，所以此结构由一个u2和一个表组成，u2代表大小
      • 索引从1开始，0代表不不引用常量池任何数据
      • 存储两大类数据
        • 字面量：字符串，final常量
        • 符号索引：包、类或接口 全限定名

    • 访问标志

    • 类索引、父类索引、接口索引集合

    • 字段表集合

    • 方法表集合

    • 属性表集合

      • code：方法的指令集合，最多不能超过65535个
      • Exception：列举所有可能抛出的受检查异常
      • LineNumberTable：Java源码行号与字节码行号之间的对应关系：可取消，取消后抛出的异常无法显示异常行号，同时也不能按照源码行进行debug
      • LocalVarialTable：局部变量表的变量与源码变量之间的关系，可取消，取消后IDE会用arg0，arg1.代替变量名

  2. 字节码指令：

    • 字节码指令=操作码+操作数
    • 面向操作数栈而不是寄存器，所以指令只有操作码，操作数放在操作数栈里
    • 操作码总量不超过255
    • 操作数无长度对其，所以超过一个字节的数据需要在运行时重建出具体的数据结构
    • 加载存储指令
    • 运算指令
    • 类型转换指令
    • 对象创建访问指令
    • 控制转移指令
    • 方法调用返回指令
    • 异常处理指令
    • 同步指令

六、虚拟机类加载机制

  1. 概述

    • 类的声明周期
      • 加载
      • 验证
      • 准备
      • 解析
      • 初始化
      • 使用
      • 卸载
    • 类加载的时机
      • 遇到new、getstatic、putstatic、invokestatic指令
        • new实例化对象
        • 获取或者设置静态变量（被final修饰、在编译期把结果放在常量池的静态字段除外）
        • 调用一个静态方法时
      • 反射调用时
      • 初始化子类时，父类没有被初始化时，需要初始化父类
      • 虚拟机启动时，初始化main方法所在的类
      • JDK7动态语言
      • JDK8，接口中有默认方法，若实现类初始化时，先初始化接口

  2. 类加载过程

    • 加载：
      • 获取二进制字节流
      • 将二进制流的静态存存储结构转化成运行时数据结构
      • 生成java.lang.Class对象
    • 验证
      • 格式验证
        • 文件格式验证：魔数、版本、编码格式utf-8
        • 元数据验证：是否有父类、父类是否允许被继承、实现类是否全部实现类抽象方法
      • 字节码验证：
        • 操作数栈里的数据和操作码匹配
        • 保证跳转指令不会跳转到方法外
      • 符号引用验证：
        • 是否能找到类、字段、方法
        • 是否允许被访问
    • 准备
      • 为类变量（static）分配内存，并且设置初始值，例如0、0.0、false等
      • final变量在编译期就赋值为设置的值
    • 解析
    • 初始化

七、字节码执行引擎

  1. 栈帧结构

    • 局部变量表

      • 存放方法参数和局部变量

      • 编译期就可以知道最大容量

      • 基本单位是32位的槽

      • 64位数据（long、double）占用两个槽，但是局部变量表属于线程私有，所以两个槽的读写不存在所谓的线程安全，不需要考虑原子性

      • 0号槽存放“this”引用

      • 槽可以复用，当PC计数器的值已经离开了某变量的作用域，则该变量的槽可被其他变量复用,但是在垃圾回收时有些许影响

      • 局部变量表属于运行时数据结构，不存在类变量在加载时的准备阶段，赋值0，0.0，false等初始值，所以必须手动赋值，否则不能用

    • 操作数栈

      • 32位
      • 大小编译期便确定了
      • 两个栈帧是独立的，出于优化处理，一个栈帧的操作数栈和上一个局部变量表部分重叠

    • 动态连接

      • 指向运行时常量池中该栈帧所属方法的引用

    • 方法返回地址

      • 正常返回：恢复上层方法的局部变量表和操作数栈，将返回值压入上层栈帧的操作数栈，PC指向调用指令的下一个指令
      • 异常：由以后处理表处理
